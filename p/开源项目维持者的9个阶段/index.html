开源项目维持者的9个阶段

The Nine Phases of an Open Source Project Maintainer

运行开源项目不只是编写代码，实际上，大多数工作都与其他事情有关。 （There is more to running an open source project than writing code. In fact most af all work has to do with something else.） 这给项目维护者带来了额外的要求，而这些要求没有被提及。在这篇文章中，我们将简要介绍9个不同的的部分，每个阶段有不同的分工。根据项目的寿命和受欢迎程度，可以将其划分为2个阶段。

Stage 1：这个项目是为你自己 大多数项目的起初要么是一个人，要么是几个人的小团队。起初做事情很简单。可以立即进行重大改变。编程语言和框架都可以改变。甚至可以毫无顾忌地转向不同的事物。由于利益相关者很少，他们通常具有相似的意识形态，因此很容易达成共识。甚至有可能完全忽略共识而“只管去做”。

Phase 1:发明家 一切始于一个想法：如何以不同的方式或更有效的方式来完成某件事。这是容易被记者甚至某些发明家所迷恋的部分。典型的叙述解释了一个天才如何设法在某个地方的地下室中独自创造出革命性的新事物。现实并不那么迷人，因为几乎所有的想法都是很多很多人已经想出来的。有些人甚至认为想法毫无价值，只有执行才重要。这有点极端，但是提出创意也是一项重要技能。

Phase 2: MVP实施者 一旦选定了一个想法，就需要编写某种原型。这是编码中最有趣的部分。这里有广阔的绿地，在那里你可以做任何事情。设计所需的体系结构，并解决有趣的问题，这些问题形成了最终产品的核心。这个阶段是人们称为程序员的主要原因。开始创造全新的事物是一种快乐的体验。不过，并不是所有的事情都是完美的，因为专注于 完成第一个版本是很重要的，而不是走弯路，最终失去兴趣。

Phase 3: 挖沟机 一旦第一个版本存在并可以使用，下一步就是使其可以用于生产。这是项目工作性质发生重大变化的地方。虽然前一阶段可以描述为有趣，但是这一阶段是乏味的。它需要面对现实世界的输入和使用情况，使最终产品可靠，流畅。这通常会暴露原始实现中需要修复或重写的bug和设计缺陷。在这个阶段很容易感到气馁，因为几天的编码结果可能与“与以前相同，但也处理了这个罕见的情况”。

工作情况（work profile）类似于用铲子挖沟。这是又脏又累又费力的工作，而且也没有多少回报。毕竟，挖了一半的沟渠和完全没挖的沟渠是一样没用。只有当你到达终点，水开始流动时，你才会有收获。物理沟渠和软件之间的区别在于，没有可靠的方法来估算你还需要挖多少。对于任何项目来说，这都是一个非常危险的阶段，因为它有可能导致资源耗尽。

Phase 4: 文档编写者 每个新项目都需要文档，但是某些项目比其他项目需要更多的文档。程序员通常不急于编写文档或使其保持最新。仅仅让用户“阅读源代码”以了解如何做事情是不够的，因为人们不想了解项目的实现细节，他们只是想使用它。有时可以让其他人来编写文档，但是通常只有在项目“变得庞大”之后才会发生。

看待文档的一种方法是，它是一种竞争优势。如果同一件事有多个竞争项目，其中一个有更好的文档，那么它获胜的几率就会更高（在其他条件相同的情况下）。编写最终用户文档需要与编写代码完全不同的方法和技能。与参考文档相比，像教程这样的文档尤其如此。

Phase 5: 营销人员 建一个更好的捕鼠器，世界会忽视你，告诉你他们捕鼠器的情况很好，非常感谢。你为什么不找一份真正的工作，而不是浪费你的时间在这个无论如何都不会工作的事情上。如果你想让他们改变主意，你需要营销。这样的例子有很多。

有许多不同的方法可以使你的项目更出名：写博客文章，在会议上进行演讲，一般的在线宣传等等。这同样需要一套新的技能，比如在公共场合与一大群人交谈。这对于内向的程序员来说尤其如此。但遗憾的是，温顺的程序员并没有继承这个世界。它往往属于那些能够对自己的解决方案发出最大声音的人。

Stage 2：该项目主要面向其他人 随着项目变得越来越大，越来越多人使用，最终会达到另一个临界点。
